두근두근 첫 스프링 프로젝트

대략적으로 10월 중순 즈음, 백엔드 및 PM 으로 새 프로젝트 안건을 맡았다.

( 라고 말하면 회사에서 일하는 것 같지만.. 사실 상 학생들 사이에서의 이벤트다. )

초기 인원 중, 나를 제외한 인원 둘(프런트 1, 백엔드 1)이 모두 개발에 직접적인 경험이 부족하여 전반적인 기획, 기술 구성, 개발의 많은 비중을

내가 짊어지고 가야했다.

데드라인까지 테스트 배포까지 성공적으로 수행했지만 아쉬운 점이 많아 어떠한 문제가 있었는 지

스스로 정리해보고자 한다.

진행하면서 느꼈던 나의 문제

1. 기획을 주도하지 못하고, 애매한 상황에서 구현단계로 진입

마감일이 한 달 남짓이라 들어 초조하고 조급했던 나는 기획을 정말 대충 구상했는데 그 때 당시 내 생각을 떠올려보자면

당장 구현할 수 있을 화면 구성과 기획내용을 노션에 TODO로 작성하고 바로 개발에 진입하는 것이었다.

위 사진 자료는 거의 2시간만에 대충 구현된 피그마 화면 구성 작업물이다.

그 때 당시의 나로서는 가장 적은시간안에 모두가 공통적으로 이해할 수 있는 최대 효율의 작업이 이것이라고 생각했나보다.

지금와서는 후회하고 왜그런 선택을 했을 까 여전히 모르겠지만,

충분한 커뮤니케이션과 상세한 기획 문서가 따로 없어도 당연히 구현해야 할 만한 것들을 팀원에게 맡기고 상세한 것은 내가 확인하고 구축하는게 효율적이라 판단했나 보다.

팀 협업 관점에서도 개발자 관점에서도 완벽한 오판이다.

초기 인원이었던 세 명이서 개발하는데 모두를 배려하지 못했다는 생각도 든다.

디자인도 한 대 때려주고 싶다. 저대로 구현할 생각은 없었는데 최대한 빠르게 산출물을 뽑고싶었다.

개발 과정 중에 발생한 문제 몇 가지

기둥을 제대로 마감하지못해 흔들거리는 빌딩처럼 기획이 마무리되었기 때문에 개발 과정 중 당연히

몇몇 문제가 발생했다.

1. 백엔드에서 일방적인 설계

BE와 FE가 같이 서로 간에 소통하며 api 를 이용해 기능을 구현하기 위해서는 최적의 방법에 대해

논의했어야 했는데 그러지 못해 일방적으로 BE에서 구현한 결과 UX가 모두의 기준에서 만족스럽지 못했다.

2. FE 와 BE 간에 일치하지 않는 유효성 검증

게시글 길이, 계정 정보 간 정규식 등 서로 일치하지 않았다. 너무 당연하게도..

3. FE 와 BE 서로 간 진행상황 파악 X

어디까지 구현되었는 지 어떻게 진행되고 있는 지 서로 간 커뮤니케이션이 적었다.

2. 개인적으로 적절하지못했던 기술 선택

초기 인원이 모두 파이썬 기초 실력정도를 겸하고 있어 백엔드 기술 선택에 큰 애매함이 있었다.

본인은 파이썬에 적응한 상태도 아니었으며 파이썬 백엔드에 대한 지식이 없었다.

그러나 프레임워크가 다 똑같겠다는 생각이 들고, 같이 백엔드를 개발할 팀원에게 더 익숙한 환경을

선택하는게 맞는 것 같아서 Python 진영의 Django 를 선택했다.

Python Django 는 흥미로웠는데 DRF (Django Restful Framework) 를 붙여서 사용해야 했으며

익숙하게 봐왔던 개발 방식이랑 많이 달라 좋은 경험이긴 했지만, 앞으로 파이썬으로 백엔드 할 일이 없다고 생각하니

배우고 코드를 작성할 의지가 뎌뎌졌다.

더군다나 파이썬에서 개발할 때는 다른언어를 쓰다오면 어느정도 진통을 겪어야

파이썬과 오해도 풀리고 파이썬 답게 코드를 작성한다 들었는데 그 과정이 또 힘겨웠다..

그러다가 백엔드를 같이하는 팀원이 빠지게되고 인증/인가 기능까지 모두 구현한 상태에서

큰 고민을 하게되었다.

2주 이상 개발을 한 시점에서 다음과 같은 점을 고려해서 스프링으로 갈아 치웠다.

  • 나는 스프링을 좀 더 깊게 공부하고 싶다.
    • 많은 커뮤니티에서 인정하는 인지도 높은 DI 프레임워크에 대해 배우고 싶다.
    • Spring WEB 프레임워크가 동작하는 방식, 궁극적으로 Spring을 이용해 하나의 웹 애플리케이션 흐름을 작성해보고 싶다.
  • 추 후, 내 결과물을 회사에 어필하고 싶다. 나는 자바 생태계에서 일하고 싶다.
  • 국내에서 가장 많이 쓰는 기술을 사용하며 공부하고 해당 기술관련 국내 커뮤니티 의견에 공감하고 싶다.
  • 자바 언어를 좀 더 숙련자답게 다루고 싶다. ( 그런 욕심으로 Effective Java 도 천천히 읽는 중.. )

3. 서버 API 테스트가 없다...(안정성 보장따위)

스프링에 적응하랴 React.js 코드 작성하랴 너무 정신이 없었다..ㅠㅠㅠㅠ

스스로가 중요성을 알면서 Postman 으로 몇가지 테스트만하면 괜찮겠지.. 하면서 테스트 코드를 작성하지 못했다.

그 결과는... 프런트의 삽질이랴... 늘 Api 에 대해 안내하면서 안정성을 보장하지못한다고

말할 때 마다 너무 미안했다.

Nest.js 개발할 때는 잘 작성했던 것으로 보아 사실 모든게 핑계처럼 느껴졌다.

포스트맨의 흔적들...그러나 여기서 끝...

4. 지루하게 반복되는 코드 더미들..

다음과 같은 서비스 인터페이스가 있다. 게시글을 관리하는 코드이다.

public interface PostCrudManager { SingleArticleInfoDto getPostById(Long id, String token); SingleArticleInfoDto updatePostById(String token, Long id, UpdateArticleRequestDto singleArticleInfoDto, List<CustomFile> customFiles); void deletePostById(String token, Long id); SingleArticleInfoDto createPost(String token, CreatePostRequestDto dto, List<CustomFile> files); List<SingleArticleInfoDto> getPostByTokenAndPagination(String token, Pageable pageable); List<SingleArticleInfoDto> getEntirePostByPagination(Pageable pageable, String token); List<SingleArticleInfoDto> getSearchParamByPagination(List<String> searchList, Pageable pageable, String token); }

각 메서드마다 사용자 또는 게시글을 가져오는 코드가 지루하게 반복된다.

User user = userRepository.findByEmail(jwtProvider.getUserEmailByToken(token)) .orElseThrow(UserNotFoundException::new); ... Article article = articleRepository.findById(id).orElseThrow(NotFoundResourceException::new);

해당 서비스뿐만 아니라 대부분의 서비스 인터페이스 구현체에서도 동일하게 반복된다.

해당 로직 뿐 만 아니라 동일한 소스코드에 대해서 private 메서드로 추상화해서 꺼내쓰는 것도 코드 최적화라고하기엔 조금 꺼리다.

근본적으로 클래스를 분리해서 (관심사 분리) 문제를 해결해보는 시도를 하는게 더 나을 것 같다.

구세주 등장

현업에서 프런트엔드 담당으로 활동하시는 갓 개발자님께서 중간에 합류하셨다.

기획에 미비했던 점, 설계에 구멍이 존재했던 점, 프로젝트 구조에 관해서,

초기에 모두 캐치해서 잡아주시고 데드라인까지 함께 해 주셨다.

안타까운 점은 남은 시간때문에 문제애 대한 인지만하고 개선하기엔 이미 기능이 다 나와버려서

당장은 불가능했던 점이었다.

이분이 없었으면 마감안에 요구사항을 충족하지 못했을 것이며,

BE, FE 영역을 넘나들며 개발할 수 있을거란 오만한 판단에 반성한다.

문제가 있었다면 개선해야지

인지하고 있었던 문제에 대한 몇몇 개선점이 떠오른다.

해결하고 개선해나갈 수 있다는 의지에 기쁘고 벅찬 마음으로 앞으로 다음과 같이 진행한다!

문서화

큰 틀(Jira 에서 Epic 개념..) 로 문제를 나뉘어서 문서를 작성해 한 눈에 보기좋은 템플릿에 배치했다.

앞서 다룬 대부분의 문제를 문서 정리해서 배치했다.

여기 좋은 템플릿 자료가 있었는데 폼 형태를 조금 손봐서 목적에 최적화된 구조를 유지할 수 있을 것 같다.

팀 협업 시스템 도입 및 커뮤니케이션 개선

팀 원이 업무 관련 개선 대안으로 Jira 를 도입하는게 어떻냐는 제안을 받았다.

비슷하게도 나에게 Jetbrains 사의 YouTrack 을 사용했던 경험이 있었고,

업무 정리 및 개발 일정 조율에 큰 도움이 되었던 것이 생각나서 바로 도입하기로 결정했다.

또한 일정 주기 또는 업무하는 날을 결정해서 매일 회고하기로 했다.

데일리 스크럼(Daily Scrum) 이라고도 하는데,

팀원 간 업무 현황을 추적하고 서로 간에 충분한 커뮤니케이션을 도모할 수 있는 좋은 방안인 것 같았다.

지금까지는 아슬아슬하게 조율하고 접근해왔지만 데드라인에 쫒기기만하는 자의 최후였고 스스로의 오만이었다.

나는 지금부터 팀 프로젝트를 더 팀 답게 만들어나가려한다! ( 땅! 땅! )

문제를 어느정도 해결하고 자랑할만한 리팩터링 기술과 소스코드로 다시 한 번 현재 프로젝트를 정리해보겠다.

스스로 화이팅이다.
