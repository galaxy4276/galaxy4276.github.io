Jenkins 와 Docker 를 활용한 Spring CI/CD 구축 (1)

이번 글에서는 Jenkins 와 Docker 에 대해서 전무후무한 상태에서 CI/CD 환경을 구축해본 경험과 과정에 대해서 서술하겠습니다.

글이 길어지면 분할 될 수 있어요.

CI/CD 환경을 구축하게 된 계기

😢 매번 직접 손으로 kill 하고 run..

결정적으로 “이번에 CI/CD 환경을 만들어보자” 라고 생각한 계기는 백엔드 개발자로 활동하는 프로젝트에서 테스트 시연 마감 2일 전부터 급격한 소스 코드의 작성 및 수정 때마다, 테스트 인스턴스에 접속해서 기존에 실행 중인 프로세스를 죽이고, 새로운 버전을 내려받고 빌드하고…실행하는 과정이 꽤나 많은 시간을 잡아먹었으며, 매 번 과정을 수행할 때마다 시간과 경험이 다소 유익하지 않았기 때문이었습니다.

백엔드 개발자가 CI/CD 환경을 구축할 줄 알면 멋있다.

요즘 다른 프런트, 백엔드 구분 없이 다른 모든 개발자들도 기본적으로 CI/CD 에 대한 이해와 구축 지식은 겸비하고 있다는 개인적인 생각이 들었습니다.

또한 개발자가 CI/CD 환경에 대한 이해도가 높다면, 시스템을 구축해두고 빌드와 배포에 시간을 할애할 필요 없이 소스 코드 작성에만 집중할 수 있다는 점이 메리트가 크다고 느꼈습니다

환경 구축하기

젠킨스 설치

CI/CD 환경을 구축하기 위해서는 일단 CI/CD 도구가 필요합니다.

저는 Jenkins 를 CI/CD 도구로 채택하였으므로 도커를 활용한 Jenkins 컨테이너를 설치하였습니다.

💡 **도커를 활용한 이유?** 시스템을 구축할 때 마다 매번 도구를 하나하나 설치하고 버전을 맞추며 관리하기 보단 컨테이너를 실행시킨다는 개념으로 추상화해서 관리하면 용이하기 때문 젠킨스의 경우 젠킨스와 JDK 버전을 하나의 이미지로 관리하고 설치할 수 있어 간편하다.

Docker Jenkins 설치는 다음 도커허브 공식문서를 참고해주세요.

Jenkins 를 통해 관리 할 프로젝트의 jdk 사용 버전에 따라 맞추어 받으면 될 것 같습니다.

저는 프로젝트에서 java 17 을 이용 중이기 때문에 다음과 같이 설치하였습니다.

docker pull jenkins/jenkins:lts-jdk17

해당 이미지를 성공적으로 설치한 후docker image ls 로 이미지 목록을 확인할 수 있습니다.

REPOSITORY TAG IMAGE ID CREATED SIZE mariadb #개인적으로 설치한 것 10.5 819fb13996d3 12 days ago 402MB jenkins/jenkins lts-jdk17 478ef6dbe1ea 2 weeks ago 462MB openjdk 17-jdk 5e28ba2b4cdb 7 months ago 471MB

이제 해당 이미지를 컨테이너로 수행시키기 위해 다음 명령어를 참고해주세요.

docker run -d -p {포트번호}:8080 -v {마운트할 디렉터리 경로}:/var/jenkins_home --name {컨테이너 이름} jenkins/jenkins:{설치한 버전}

포트번호 외부에서 접근할 포트 번호
마운트 할 디렉터리 경로 해당 경로로 바인드 마운트를 수행합니다.
컨테이너 이름 실행 할 컨테이너 이름을 설정합니다. ( 안정하면 랜덤 )
설치한 버전 이미지의 버전

저는 다음과 같은 명령어를 수행하였습니다.

docker run -d -p 81:8080 -v /home/galaxy4276/jenkins:/var/jenkins_home --name test-j enkins-docker jenkins/jenkins:lts-jdk17

다음과 같이 설정하고 http://localhost:{설정한 포트} 로 진입하면 젠킨스 초기화면이 등장합니다.

💡 ???? 비밀번호 입력이라니…

젠킨스 초기 설정을 수행하기 위해 처음 실행되면서 설정된 패스워드를 입력해야 합니다.

화면에 다음과 같은 경로로 진입하면 해당 패스워드 파일이 존재합니다.

/var/jenkins_home/secrets/initialAdminPassword

실행 중인 도커 컨테이너에 bash 로 진입해서 확인해도 되지만, 사전에 호스트 OS 디렉터리 경로에 마운트를 수행하였기 때문에 해당 경로에서 진입하시면 편합니다.

💡 컨테이너에 직접 접근해서 확인하실 분들은 다음 명령어 참고 바랍니다.

docker exec -it {컨테이너 이름 또는 id} /bin/bash

전 기존에 홈 디렉터리에 마운트를 수행하였기 때문에 다음과 같은 명령으로 확인하였습니다.

초기 패스워드를 입력하고 나면 설치화면이 나오는데 추천 플러그인 설치 선택하시면 됩니다.

설치 과정을 마친 이후 계정 설정까지 진행합니다.

이후로 URL 을 설정하고 ( 기본 값으로 설정 )

Start using Jenkins 를 눌러 초기 설정을 마칩니다.

Jenkins 와 Github 키 등록하기

Git 으로 관리 중인 프로젝트를 Jenkins 와 연동하기 위해 공개 키 암호시스템을 활용하여 키를 생성하겠습니다.

💡 생성한 키는 추 후 자동 배포할 때도 활용할 예정입니다.

저는 기존에 마운트한 jenkins 경로에 .ssh 디렉터리를 생성해서 작업하겠습니다.

/home/galaxy4276/jenkins/.ssh

다음 명령어를 활용해서 공개 키, 개인 키를 생성하였습니다.

💡 공개 키, 개인 키에 대한 내용은 [링크](https://blog.naver.com/PostView.naver?blogId=chodahi&logNo=221385524980)를 참고하세요.

💡 passphrase 입력을 요구하는 것은 입력한 평문을 활용해 개인 키를 암호화 하며, 권장 값은 10 ~ 30 문자라고 합니다. 저는 일회용으로 만드는거라 해당 과정을 생략하였습니다

우선 생성한 공개 키를 Github 에 등록하겠습니다.

개인 Github 페이지에서 Settings - SSH and GPG Keys 항목으로 진입해주세요.

New SSH Key 버튼을 눌러 공개 키를 등록해주세요.

여기서 개인 키(id_rsa)가 아닌 공개 키(id_rsa.pub) 입니다!

Jenkins 에 프로젝트 등록하기

젠킨스 메인 좌측에서 새로운 Item 을 눌러 Freestyle Project 를 생성합니다.

이름은 아무렇게나 지어도 좋습니다.

필수과정은 아니지만, 저는 github project 를 체크하고 레포 주소를 업로드하였습니다.

Git 과 연동하기 위한 중요한 부분입니다.

현재 설정에서 Repository URL은 https 주소로 넣으면 안되고, ssh 주소로 기입해야 합니다.

아마 주소를 기입하면 에러가 날텐데 사전에 생성한 개인 키를 이용해서 자격증명(Credentials) 를 등록해야 합니다.

+Add - Jenkins 를 눌러주세요.

필드 참고해주세요.

Kind - SSH Username with private key 로 설정해야함

ID 는 docker 컨테이너의 사용자 이름이며 기본 값은 jenkins 입니다.

Username 은 젠킨스에서 식별하기 위한 값으로 임의로 지정하시면 됩니다.

Private Key 항목은 Enter directly 를 선택하고 기존에 생성했던 개인 키(id_rsa) 내용을 전부 기입하세요.

💡 개인 키는 외부로 공개하면 안됩니다.
# 개인 키 예시 -----BEGIN OPENSSH PRIVATE KEY----- b3BlbnNzaC1rZXktdjEAAAAABG5vbmUAAAAEbm9uZQAAAAAAAAABAAABlwAAAAdzc2gtcn NhAAAAAwEAAQAAAYEAtCv8qtQbvxfRC33vmDdFyltJW4PzAIISmsmGlle+QYngL1QPl67j ...생략 -----END OPENSSH PRIVATE KEY-----

기존에 비밀번호(passphrase) 를 설정하셨다면 아래 옵션에도 해당 값을 기입하고 Add 합니다.

자격증명을 부여해도 다음과 같은 에러가 난다!

해당 에러는 github.com 을 ssh 키 통신의 known_host 항목에 추가해야 한다는 오류입니다.

처음 접속하는 ssh 입장에서 해당 호스트로 접근해도 되는 지에 대한 등록이 되어있지 않기 때문에 발생하는 오류이며 해결방법은 간단하니 안심하셔도 됩니다.

저는 다음과 같은 방법으로 해결했으며 known_hosts 파일에 직접 등록해서 해결해도 될 것 같습니다.

  1. 해당 도커 컨테이너에 bash 셸 접근
  2. 에러에 나온 명령어를 입력해서 known_hosts 에 등록

known_hosts 등록하는 과정(야매) 을 수행하고 나서는 에러가 등장하지 않고 정상적으로 등록이 수행된 것을 알 수 있습니다.

빌드에 사용할 브랜치 네임 패턴을 입력해줍니다.

Git 에 commit 을 수행할 때 마다 젠킨스에 빌드 알림을 주기 위해 해당 옵션을 체크합니다.

Github Webhook 설정하기

프로젝트 리포지토리에서 Settings - Webhooks 항목으로 들어갑니다.

이 후 Payload URL 에 젠킨스 주소/github-webhook/ 을 설정하고,

Content Type 은 application/json 으로 설정해서 webhook 을 만들어주시면 되는데 한 가지 문제사항이 있습니다.

위와 같은 환경에서 Github (외부망)에서 [localhost:8](http://localhost:81)0(내부망) 으로 네트워크 요청이 들어올 수 있게 설정해야 하기 때문입니다.

저같은 경우는 iptime 공유기를 사용하고 있으므로, ddns 을 설정하고 포트포워딩 정책을 걸어서 접근할 수 있게하였고 구글링 결과 많은 분들이 ngrok 서비스를 이용해서 외부에서 접근할 수 있게 설정하시는 것 같습니다.

그러니 이 부분은 개인 환경을 고려해서 설정해주시면 될 것 같습니다.
